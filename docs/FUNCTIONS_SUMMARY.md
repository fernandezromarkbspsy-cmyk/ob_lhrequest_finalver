## Project Purpose

This project is a separated Go/Echo API backend and static frontend linehaul dashboard with GORM/Postgres persistence and vanilla JavaScript for client-side request tables, modals, filtering, notifications, and role-specific actions.

## Main Workflow

- Ops PIC and FTE Ops create linehaul requests.
- New requests start as Pending (PENDING) and appear across request views.
- FTE Ops can edit, cancel, or approve requests.
- Approved requests route to FTE MM as Approved (APPROVED).
- FTE MM can reject back to FTE Ops, assign the truck, then assign plate details to move the request to For Docking.
- Dock Officer / Doc Officer handles For Docking requests by entering driver ID, LHTrip number, and docking time.
- Queue counts, notifications, and table statuses are shared across roles through /api/stats and /api/requests.

## Runtime Entry Points

- cmd/server/main.go: backend API startup, DB connection, GORM migration, workflow status constraint migration, Echo setup, CORS, and route registration.
- cmd/frontend/main.go: local static frontend server for the rontend directory.

## Database Models

- Cluster: cluster lookup data used by the New LH Request modal for cluster, region, dock, and backlogs mapping.
- Request: linehaul request lifecycle data, including status, truck details, plate number, driver ID, LHTrip number, and docking timestamps.
- User: role-aware login identity for ops_pic, te_ops, te_mm, and dock_officer / doc_officer.

## Status Values

- PENDING: shown as Pending; FTE Ops queue.
- APPROVED: shown as Approved; FTE MM queue.
- ASSIGNED: shown as Assigned; plate assignment step.
- FOR_DOCKING: shown as For Docking; Dock Officer queue.
- DOCKED: docked truck awaiting confirmation.
- CONFIRMED: completed request.
- REJECTED_BY_MM: rejected by FTE MM and routed back to FTE Ops.
- CANCELLED: canceled request.
